# Plan Code

## Purpose

Plan the technical implementation before writing any application code. This skill
covers solution architecture, framework decisions, connector planning, coding
standards, error handling strategy, and the testing approach — everything needed
so that `implement-code` can execute without ambiguity.

## Deliverable

A **Technical Implementation Plan** document covering architecture decisions,
connector manifest, coding standards, error handling patterns, and test strategy.

## Architecture Decisions

### Framework Choice

| Option | When to choose |
|--------|---------------|
| React | Default. Largest ecosystem, best Copilot support, most community samples |
| Vue | Team prefers Vue. Good Copilot support |
| Vanilla TS | Minimal overhead, no framework lock-in, smaller bundle |

### State Management

| Option | When to choose |
|--------|---------------|
| React Context + hooks | Simple apps, <10 data entities |
| Zustand / Jotai | Medium complexity, shared state across many components |
| Redux Toolkit | Large apps, complex async flows, team already uses Redux |

### Component Library

| Option | When to choose |
|--------|---------------|
| Fluent UI React | Default. Microsoft's design system, natural fit for Power Platform |
| Shadcn/UI | Modern, composable, Tailwind-based |
| MUI | Team already uses it, rich component set |

### Routing

| Option | When to choose |
|--------|---------------|
| React Router | Multi-page SPA with distinct views |
| Single view | Dashboard or form-based app with tabs/panels |

## Connector Manifest

Map every data requirement to a specific Power Platform connector.

### Connector Planning Template

For each connector the app needs:

```markdown
### Connector: [Name]

| Field | Value |
|-------|-------|
| API Name | e.g., shared_commondataserviceforapps |
| Type | Standard / Premium / Custom |
| Connection ID | [from pac connection list] |
| DLP Group | Business / Non-Business / Blocked |
| Operations Used | e.g., GetItems, CreateItem, UpdateItem, DeleteItem |
| Tabular? | Yes (requires table ID and dataset) / No (action-based) |
```

### DLP Compatibility Check

Before finalising the connector list:

1. List DLP policies on the target environment (Admin Center → Data policies)
2. Check which group each connector falls into
3. **Rule**: Business group connectors cannot share data with Non-Business group
4. If conflict exists: request exception, separate into two apps, or use Power Automate as middleware

## Coding Standards

### TypeScript Configuration

- `strict: true` — non-negotiable
- `noUncheckedIndexedAccess: true` — forces handling undefined
- Path aliases for clean imports (`@components/`, `@hooks/`, `@services/`)
- `src/generated/` excluded from linting (auto-generated code)

### Linting & Formatting

- **ESLint**: `@typescript-eslint/strict-type-checked` + `jsx-a11y/recommended`
- **Prettier**: single quotes, trailing commas, 90 char width
- **Husky**: pre-commit hook runs lint-staged
- **jsx-a11y plugin**: catches accessibility violations at lint time

### Naming Conventions

| Item | Convention | Example |
|------|-----------|---------|
| Components | PascalCase | `AccountList.tsx` |
| Hooks | camelCase with `use` | `useAccounts.ts` |
| Utils | camelCase | `formatCurrency.ts` |
| Types | PascalCase | `Account.ts` |
| Tests | `[source].test.tsx` | `AccountList.test.tsx` |
| Constants | UPPER_SNAKE_CASE | `MAX_PAGE_SIZE` |
| Branches | `feature/[ticket]-description` | `feature/42-add-filters` |
| Commits | Conventional Commits | `feat:`, `fix:`, `docs:` |

## Error Handling Strategy

### Error Layers

1. **Global Error Boundary** — catches unhandled React rendering errors
2. **Connector Wrapper** — wraps every generated service call with retry logic and error normalisation
3. **Hook-Level State** — every data hook exposes `{ data, isLoading, error, refetch }`
4. **UI Error Components** — standardised `ErrorBanner`, `ErrorPage`, `Toast` components

### Retry Policy

- Retryable status codes: 408, 429, 500, 502, 503, 504
- Max retries: 2
- Backoff: exponential (1s, 2s)
- Non-retryable: 400, 401, 403, 404

### Logging

- Logger abstraction with pluggable transports
- Console transport in development
- Application Insights transport in production (optional)

## Test Strategy

### Test Layers

| Layer | Tool | What to test | Coverage target |
|-------|------|-------------|-----------------|
| Unit | Jest + RTL | Components, hooks, utils | 80% lines |
| Integration | RTL + MSW | Navigation flows, form submissions | Critical paths |
| E2E | Playwright | Auth flow, top 5 user journeys | Happy paths |
| Accessibility | axe-core + jest-axe | Every component | Zero violations |
| UAT | Manual | All capabilities from project brief | Sign-off |

### Key Testing Patterns

- Mock generated services with `jest.mock()` — never call real connectors in unit tests
- Use MSW (Mock Service Worker) for integration tests
- Pre-authenticate Playwright with stored auth state
- Run `jest-axe` on every component to catch accessibility regressions
- UAT scripts provided to testers with step-by-step instructions

| CI/CD | GitHub Actions or Azure Pipelines (optional) |
| Solution backup | `pac solution export` before each release |

## Security Checklist

- [ ] No API keys or secrets in source code
- [ ] Auth handled entirely by Power Apps SDK (no custom auth)
- [ ] User permissions enforced at connector level
- [ ] DLP compliance verified
- [ ] Input validation on all form fields
- [ ] No `dangerouslySetInnerHTML` without sanitisation
- [ ] HTTPS only (handled by Power Platform hosting)
