# Phase 5: Connectors & Data Sources

## Purpose

Translate the data architecture (Phase 3) into a concrete connector manifest: which
Power Platform connectors to use, how to add them to the project, and how to structure
connection references for multi-environment ALM. This phase gates Phase 6 (Scaffolding).

## Deliverable

A **Connector Manifest** — a table of every connector needed, with API name, type,
DLP group, connection reference name, and the exact `pac code add-data-source` commands.

## Unsupported Connectors

These connectors cannot be used in code apps (as of initial release):

- **Excel Online (Business)** (`shared_excelonlinebusiness`)
- **Excel Online (OneDrive)** (`shared_excelonline`)

If Excel was planned in Phase 3, substitute with Dataverse or SharePoint Lists now.

## DLP Compatibility Check

Before finalising the manifest, verify DLP compliance:

1. List DLP policies on the target environment (Admin Center → Data policies)
2. Check each connector's group assignment
3. **Rule**: connectors in the Business group cannot share data with connectors in the Non-Business group
4. If conflict: request exception, split the app, or route data via Power Automate

## Adding Data Sources

Always use connection references for portability across Dev/Test/Prod environments.

### Step 1: Discover

```bash
# List all connections
pac connection list

# Discover datasets (for tabular sources)
pac code list-datasets -a <apiName> -c <connectionId>

# Discover tables within a dataset
pac code list-tables -a <apiName> -c <connectionId> -d <datasetName>

# List connection references in your solution
pac code list-connection-references -env <environmentURL> -s <solutionID>
```

### Step 2: Add via connection reference (recommended)

```bash
# Action-based connector
pac code add-data-source \
  -a <apiName> \
  -cr <connectionReferenceLogicalName> \
  -s <solutionID>

# Tabular connector
pac code add-data-source \
  -a <apiName> \
  -cr <connectionReferenceLogicalName> \
  -s <solutionID> \
  -t <tableId> \
  -d <datasetName>
```

### Step 3: Verify generated files

After each `add-data-source` command, verify files were created:

```
generated/
└── services/
    ├── <ConnectorName>Model.ts    ← data types (do not edit)
    └── <ConnectorName>Service.ts  ← service methods (do not edit)
```

**Never edit files in `generated/services/`** — they are regenerated by the CLI.
Commit them to source control.

### Schema change workflow

If a connector's schema changes after initial setup:

```bash
# Delete the existing data source
pac code delete-data-source -a <apiName> -ds <dataSourceName>

# Re-add with the updated schema
pac code add-data-source -a <apiName> -cr <connectionReferenceLogicalName> -s <solutionID> ...
```

## Using Generated Services in Code

Generated services expose typed methods that map to connector operations:

```typescript
// Import generated service and model
import { DataverseService } from './generated/services/DataverseService';
import type { Account } from './generated/models/DataverseModel';

// Action-based usage
const profile = await Office365UsersService.MyProfile_V2("id,displayName,jobTitle");

// Tabular CRUD
const accounts = await AccountsService.getall();
const account  = await AccountsService.get(id);
const newId    = await AccountsService.create(record);
await AccountsService.update(id, changedFields);
await AccountsService.delete(id);
```

> Keep the relevant `*Service.ts` and `*Model.ts` files open in VS Code when
> prompting GitHub Copilot — it will use the exact method signatures and types.

## State Management Patterns

### The hook pattern

Each connector gets a wrapper hook that exposes a consistent interface. This prevents
redundant connector calls across components and makes loading/error states composable.

```typescript
// src/hooks/useAccounts.ts
import { useState, useEffect, useCallback } from 'react';
import { AccountsService } from '../generated/services/AccountsService';
import type { Account } from '../generated/models/DataverseModel';

interface UseAccountsResult {
  data: Account[] | null;
  isLoading: boolean;
  error: Error | null;
  refetch: () => void;
}

export function useAccounts(): UseAccountsResult {
  const [data, setData] = useState<Account[] | null>(null);
  const [isLoading, setIsLoading] = useState(true);
  const [error, setError] = useState<Error | null>(null);

  const fetch = useCallback(async () => {
    setIsLoading(true);
    setError(null);
    try {
      const result = await AccountsService.getall();
      setData(result.value ?? []);
    } catch (err) {
      setError(err instanceof Error ? err : new Error('Unknown error'));
    } finally {
      setIsLoading(false);
    }
  }, []);

  useEffect(() => { void fetch(); }, [fetch]);

  return { data, isLoading, error, refetch: fetch };
}
```

### Avoiding redundant connector calls

Lift shared hooks to the nearest common ancestor rather than calling the same hook
in multiple child components — React re-renders cascade downward so data fetched at
the parent level is available to all children without extra calls.

```typescript
// Parent component — fetch once, pass down
function AccountsPage() {
  const { data: accounts, isLoading, error } = useAccounts();
  return (
    <>
      <AccountSummaryCard accounts={accounts} isLoading={isLoading} />
      <AccountList accounts={accounts} isLoading={isLoading} error={error} />
    </>
  );
}
```

### Cache invalidation after mutations

After a create, update, or delete operation, call `refetch` from the parent hook to
keep the list in sync without a full page reload:

```typescript
function AccountsPage() {
  const { data, isLoading, error, refetch } = useAccounts();

  const handleDelete = async (id: string) => {
    await AccountsService.delete(id);
    refetch(); // re-fetch the list after deletion
  };

  return <AccountList accounts={data} onDelete={handleDelete} />;
}
```

For more complex state (optimistic updates, cross-page cache sharing), consider
[Zustand](https://zustand-demo.pmnd.rs/) — see `agents/skills/plan-code/SKILL.md`
for the state management comparison table.
