# Phase 9: Governance, Handover & Documentation

## Purpose

Ensure the code app is documented, governable, and supportable after the
development team moves on. This phase produces the artefacts that a support
team, future developer, or client admin needs to operate, maintain, and
evolve the app.

## Deliverable

A **Handover Pack** containing: technical documentation, operational runbook,
admin guide, and a governance checklist.

## Why This Phase Exists

Code apps are new to most Power Platform teams. Unlike canvas apps where the
maker portal is the IDE and the app is self-describing, code apps live in a
Git repo with a build process. The support team needs clear documentation to
avoid the "only the original developer can touch it" trap.

## Getting Started

### Clone and install
```bash
git clone [repo-url]
cd [app-name]
npm install
```

### Authenticate to Power Platform
```bash
pac auth create --environment [environment-url]
```

### Run locally
```bash
npm run start
```

### Run tests
```bash
npm test                # Unit + integration tests
npm run test:e2e        # End-to-end tests (requires auth)
```

### Deploy
```bash
npm run build
pac code push --solution-unique-name [solution-name]
```

### 2. Operational Runbook

A runbook for the support team covering day-to-day operations:

```markdown
# Operational Runbook: [App Name]

## Monitoring
- **Health metrics**: Power Platform Admin Center → Apps → [app name] → Health
- **Application errors**: [describe where errors surface — console, App Insights, etc.]

## Common Issues & Resolutions

### App fails to load
1. Check Power Platform service health: https://admin.powerplatform.microsoft.com
2. Verify code apps feature is still enabled on the environment
3. Check DLP policies haven't changed to block a connector
4. Check user has Power Apps Premium licence assigned

### Connector errors
1. Verify the connection hasn't expired in Power Apps → Connections
2. Check if the user has permissions on the underlying data source
3. Review DLP policies for connector grouping conflicts

### Deployment fails
1. Verify `pac auth` is active: `pac auth list`
2. Check Node.js version matches prerequisites
3. Ensure the solution exists in the target environment
4. Review build output for TypeScript errors

## Routine Maintenance
| Task | Frequency | Procedure |
|------|-----------|-----------|
| Review connection health | Monthly | Power Apps portal → Connections, refresh any expired |
| Update npm dependencies | Quarterly | `npm audit`, `npm update`, run test suite, redeploy |
| Review user access | Quarterly | Power Apps portal → app sharing, remove leavers |
| Backup solution | Before each release | `pac solution export` |
| Review DLP policy alignment | After any DLP change | Compare connector manifest against updated policy |

## Escalation Path
| Level | Contact | Response Time |
|-------|---------|--------------|
| L1 — App access, permissions | [support team] | 4 hours |
| L2 — Bug fix, data issue | [dev team] | 1 business day |
| L3 — Architecture change, new feature | [lead dev / consulting] | Planned sprint |

## Rollback Procedure
1. Identify the last known good solution version in source control
2. Checkout that version: `git checkout [tag/commit]`
3. Build: `npm run build`
4. Deploy: `pac code push --solution-unique-name [solution]`
5. Verify in Power Apps portal
```

## Governance Checklist

Before handover, verify:

### Code Quality
- [ ] No `console.log` statements in production code
- [ ] No hardcoded connection IDs, URLs, or secrets
- [ ] TypeScript strict mode enabled
- [ ] ESLint configured with consistent ruleset
- [ ] Prettier or equivalent formatter configured
- [ ] All tests passing
- [ ] Test coverage meets thresholds (≥70% branches, ≥80% lines)

### Security
- [ ] No API keys or secrets in source code
- [ ] Authentication handled entirely by Power Apps SDK (no custom auth code)
- [ ] User permissions verified at connector level
- [ ] DLP compliance confirmed
- [ ] Input validation on all form fields (client-side and connector-level)

### Operations
- [ ] README.md complete and accurate
- [ ] Operational runbook reviewed by support team
- [ ] Admin guide reviewed by Power Platform admin
- [ ] Monitoring/alerting configured
- [ ] Rollback procedure tested
- [ ] Solution exported and committed to source control

### Compliance
- [ ] Accessibility compliance report complete (Phase 8)
- [ ] Data protection impact assessment (DPIA) filed if required
- [ ] App registered in organisational app catalogue
- [ ] Retention and data handling policies documented

## Naming Conventions Standard

Apply these conventions across all code app projects for consistency:

### Solution and App
- Solution name: `[ClientCode]_[AppName]` e.g., `ACME_AssetTracker`
- App display name: Human-readable, e.g., "Asset Tracker"
- Repo name: kebab-case, e.g., `acme-asset-tracker`

### Code
- Components: PascalCase, e.g., `AccountList.tsx`
- Hooks: camelCase with `use` prefix, e.g., `useAccounts.ts`
- Utils: camelCase, e.g., `formatCurrency.ts`
- Types: PascalCase, e.g., `Account.ts`
- Tests: `[source].test.tsx`
- Constants: UPPER_SNAKE_CASE

### Dataverse (if applicable)
- Publisher prefix: consistent per client, e.g., `acme_`
- Table names: lowercase with prefix, e.g., `acme_asset`
- Column names: lowercase with prefix, e.g., `acme_serialnumber`

### Git
- Branch naming: `feature/[ticket-id]-short-description`
- Commit messages: Conventional Commits (`feat:`, `fix:`, `docs:`, `test:`)
- Tags: Semantic versioning `v1.0.0`
