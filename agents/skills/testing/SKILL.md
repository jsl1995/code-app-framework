# Phase 7: Testing & Quality Assurance

## Purpose

Define and execute a testing strategy that covers unit tests, integration tests,
end-to-end tests, and user acceptance testing for code apps. Because code apps are
standard SPAs, you have access to the full web testing ecosystem — unlike canvas apps
which are limited to Power Apps Test Studio.

## Deliverable

A **Test Plan** document and a working test suite integrated into the project.

## Testing Layers

### Layer 1: Unit Tests

Test individual functions, hooks, and utility logic in isolation.

**Tools**: Jest + React Testing Library (default), Vitest as alternative

**What to test**:
- Custom hooks (data fetching, state logic)
- Utility functions (formatting, validation, calculations)
- Component rendering (correct elements appear given props/state)
- Conditional rendering (loading, error, empty states)

**What NOT to test**:
- Generated service files (these are auto-generated by the SDK — trust them)
- The Power Apps SDK internals
- Third-party library internals

**Pattern — mocking generated services**:
```typescript
// __mocks__/generated/services/DataverseService.ts
export const DataverseService = {
  getAccounts: jest.fn(),
  createAccount: jest.fn(),
  updateAccount: jest.fn(),
  deleteAccount: jest.fn(),
};

// In your test file
import { DataverseService } from '../generated/services/DataverseService';
import { render, screen, waitFor } from '@testing-library/react';
import { AccountList } from './AccountList';

jest.mock('../generated/services/DataverseService');

test('displays accounts after loading', async () => {
  (DataverseService.getAccounts as jest.Mock).mockResolvedValue([
    { id: '1', name: 'Contoso Ltd', statecode: 0 },
  ]);

  render(<AccountList />);
  
  expect(screen.getByTestId('loading-spinner')).toBeInTheDocument();
  
  await waitFor(() => {
    expect(screen.getByText('Contoso Ltd')).toBeInTheDocument();
  });
});

test('shows error state when fetch fails', async () => {
  (DataverseService.getAccounts as jest.Mock).mockRejectedValue(
    new Error('Connector unavailable')
  );

  render(<AccountList />);
  
  await waitFor(() => {
    expect(screen.getByText(/something went wrong/i)).toBeInTheDocument();
  });
});
```

**Naming convention**: `[ComponentName].test.tsx` or `[hookName].test.ts`
co-located alongside the source file.

### Layer 2: Integration Tests

Test that multiple components work together correctly, including routing
and state management.

**Tools**: React Testing Library with `MemoryRouter`, MSW (Mock Service Worker)
for API mocking

**What to test**:
- Navigation flows (list → detail → edit → save → back to list)
- Form submission → data refresh cycle
- Filter/search → result update
- Error boundary catching downstream failures

**Pattern — MSW for connector mocking**:
```typescript
import { setupServer } from 'msw/node';
import { rest } from 'msw';

const server = setupServer(
  rest.get('/api/connector/accounts', (req, res, ctx) => {
    return res(ctx.json([
      { id: '1', name: 'Contoso Ltd' },
      { id: '2', name: 'Fabrikam Inc' },
    ]));
  })
);

beforeAll(() => server.listen());
afterEach(() => server.resetHandlers());
afterAll(() => server.close());
```

### Layer 3: End-to-End Tests

Test the full app running in a browser against the real (or test) Power Platform
environment.

**Tools**: Playwright (recommended) or Cypress

**Important considerations**:
- Code apps require Entra ID authentication — you need a test account or
  a way to handle the auth flow in your E2E setup
- Use a dedicated test environment with known test data
- E2E tests are slower and more brittle — keep the suite focused on critical paths

**What to test E2E**:
- Authentication flow (consent dialog on first launch)
- The 3–5 most critical user journeys from the project brief
- Cross-browser rendering (Edge, Chrome at minimum)
- Responsive behaviour at mobile breakpoint

**Pattern — Playwright with auth**:
```typescript
import { test, expect } from '@playwright/test';

test.describe('Critical Path: Create and View Item', () => {
  test.use({ storageState: 'auth-state.json' }); // Pre-authenticated state

  test('user can create a new item and see it in the list', async ({ page }) => {
    await page.goto('/items/new');
    await page.fill('[data-testid="item-name"]', 'Test Item');
    await page.fill('[data-testid="item-description"]', 'Created by E2E test');
    await page.click('[data-testid="submit-button"]');
    
    await expect(page).toHaveURL(/\/items\/[a-f0-9-]+/);
    await expect(page.locator('h1')).toContainText('Test Item');
  });
});
```

### Layer 4: User Acceptance Testing (UAT)

UAT is performed by real users, not automated. Provide a structured framework.

**UAT preparation checklist**:
- [ ] Deploy to a dedicated UAT environment
- [ ] Prepare test accounts with appropriate security roles
- [ ] Create test data that represents realistic scenarios
- [ ] Write UAT scripts covering each capability from the project brief
- [ ] Provide a feedback template (see below)
- [ ] Schedule UAT sessions with diverse users (different roles, devices, IT comfort)

**UAT script template**:

## Performance Testing

Code apps load inside the Power Apps player, so there's inherent overhead.
Test for acceptable performance:

- **Initial load time**: Target <3 seconds on corporate network
- **Navigation between views**: Target <1 second
- **Data-heavy views**: Test with realistic row counts, ensure pagination works
- **Bundle size**: Run `npm run build` and check the output size. Aim for <500KB gzipped

**Tools**: Lighthouse (built into Chrome DevTools), Web Vitals

## Test Configuration

### jest.config.ts (starter)
```typescript
export default {
  preset: 'ts-jest',
  testEnvironment: 'jsdom',
  setupFilesAfterSetup: ['./src/test-setup.ts'],
  moduleNameMapper: {
    '^@/(.*)$': '<rootDir>/src/$1',
    // Mock generated services globally
    '^.*/generated/services/(.*)$': '<rootDir>/src/__mocks__/generated/services/$1',
  },
  collectCoverageFrom: [
    'src/**/*.{ts,tsx}',
    '!src/generated/**',  // Exclude auto-generated files
    '!src/**/*.d.ts',
  ],
  coverageThresholds: {
    global: {
      branches: 70,
      functions: 70,
      lines: 80,
      statements: 80,
    },
  },
};
```

### package.json scripts
```json
{
  "scripts": {
    "test": "jest",
    "test:watch": "jest --watch",
    "test:coverage": "jest --coverage",
    "test:e2e": "playwright test",
    "test:e2e:headed": "playwright test --headed"
  }
}
```

## GitHub Copilot Prompts

**Generate unit tests for a component**:
```
@workspace Generate unit tests for #file:src/components/AccountList.tsx using
Jest and React Testing Library. Mock the DataverseService. Test: loading state,
successful data display, error state, empty state, and filter interaction.
Follow the naming convention [Component].test.tsx.
```

**Generate E2E test for a user journey**:
```
@workspace Write a Playwright E2E test for this user journey: User navigates
to the dashboard, clicks on a recent item, edits the name field, saves, and
verifies the name is updated. Use data-testid attributes for selectors.
Assume authentication is handled via stored state.
```

**Generate test data factory**:
```
@workspace Create a test data factory in src/test-utils/factories.ts that
generates realistic mock data for each entity in #file:src/types/. Use a
builder pattern so tests can override specific fields. Include a helper
to generate arrays of N items.
```

## Transition

Testing is ongoing throughout the lifecycle. After initial test suites are
in place, proceed to Phase 8 (Accessibility & Compliance) or Phase 9
(Governance & Handover) as needed.
